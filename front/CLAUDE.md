# CLAUDE.md

이 문서는 이 프로젝트에서 AI 코딩 도구가 반드시 따라야 할 작업 규칙이다.
사용자는 프론트엔드에 익숙하지 않으므로, 긴 설명이나 차이점 표보다 실제 코드 수정과 화면 확인을 우선한다.

## 1. 프로젝트 기본 정보

- 프레임워크: React 19 + Vite
- 라우팅: react-router-dom 6
- 스타일: Tailwind v4, CSS-first config
- 상태관리: Context API
- HTTP: axios
- 배포: Vercel
- 앱 형태: 모바일 PWA
- 새 npm 패키지 추가 금지

## 2. 폴더 역할

```text
src/
├── main.jsx              # 진입점. 수정하지 말 것.
├── App.jsx               # 라우팅 테이블. 새 페이지 추가가 아니면 수정하지 말 것.
├── index.css             # 디자인 토큰, 전역 스타일, app-shell
├── pages/                # 화면 단위 컴포넌트
├── components/           # 재사용 UI 컴포넌트
├── api/                  # 백엔드 통신. 시각 작업과 무관하면 수정 금지.
└── context/              # 전역 상태. 시각 작업과 무관하면 수정 금지.
```

`App.css`는 Vite 템플릿 잔재로 보고, import 되어 있지 않으면 무시한다.

## 3. 작업 원칙

- 사용자가 Figma URL을 주면 MCP로 디자인 정보를 읽고 바로 코드에 반영한다.
- 사용자가 명시적으로 요청하지 않는 한 차이점 표를 만들지 않는다.
- 라우팅, API, 인증, 상태 로직처럼 앱 동작에 영향 있는 변경이 필요할 때만 먼저 질문한다.
- 시각 변경은 바로 진행한다.
- 한 번에 전체 앱을 바꾸려 하지 말고, 한 화면 또는 한 컴포넌트 단위로 수정한다.
- 수정은 최소 범위로 한다. 요청과 직접 관련 없는 리팩터링, 정리, 파일 이동은 하지 않는다.
- 기존 스타일과 구조를 우선 따른다.

## 4. Figma 이식 규칙

Figma MCP가 생성한 JSX를 그대로 복사하지 않는다.
Figma는 정적인 캔버스이고, 실제 앱은 여러 휴대폰 폭에서 동작해야 한다.

Figma에서 참고할 것:

- 구조
- padding
- gap
- icon size
- font size
- line-height
- color
- radius
- shadow
- active/inactive 상태
- SVG path

Figma에서 그대로 복사하면 안 되는 것:

- absolute left/top 좌표
- 캔버스 고정 width
- 불필요한 wrapper div
- 외부 asset URL에 의존하는 아이콘
- MCP가 만든 과도한 className 조합

## 5. 모바일 레이아웃 기준

이 앱은 모바일 PWA이며, 기준 최대 폭은 448px이다.

앱 셸 기준:

```css
width: 100%;
max-width: 448px;
margin: 0 auto;
```

폭 처리 규칙:

- `w-[393px]`, `w-[430px]`, `w-[448px]` 같은 화면 고정 폭은 피한다.
- 화면 전체를 채우는 요소는 `w-full max-w-[448px]` 패턴을 사용한다.
- 내부 배치는 absolute 대신 flex, grid, gap, padding, flex-1을 우선 사용한다.
- Figma의 393px은 디자인 캔버스 기준일 뿐, 코드의 고정 폭 기준이 아니다.
- 360px, 393px, 430px, 448px 폭에서 깨지지 않아야 한다.

BottomTabBar 표준 패턴:

```text
fixed bottom-0 left-1/2 -translate-x-1/2 w-full max-w-[448px]
```

페이지 콘텐츠는 BottomTabBar에 가리지 않도록 충분한 하단 padding을 가져야 한다.

## 6. 디자인 토큰

디자인 토큰은 `src/index.css`의 `@theme inline`에 정의한다.

브랜드 컬러는 JSX에 하드코딩하지 않는다.

```text
bg-[#5B40E4]      -> bg-primary
text-[#5B40E4]    -> text-primary
border-[#5B40E4]  -> border-primary
```

규칙:

- 브랜드 hex 색상은 토큰 클래스로 바꾼다.
- Figma에 꼭 필요한 색이 기존 토큰에 없으면 `index.css`에 새 토큰을 추가한다.
- Tailwind 기본 gray 계열은 사용해도 된다.
- 단, 브랜드 컬러와 헷갈리는 회색/보라색은 토큰화 여부를 검토한다.

## 7. 절대 건드리지 말 것

시각 작업 중 아래 항목은 수정하지 않는다.
수정이 꼭 필요해 보이면 사용자에게 먼저 물어본다.

- `App.jsx` 라우팅 테이블
- 각 페이지와 컴포넌트의 `navigate('/...')` 경로 문자열
- 인증 로직
- API 호출
- 폼 제출 로직
- 상태 변경 로직
- `useNavigate`, `useLocation`, `useAuth`, `useEffect` 같은 기존 hook 사용 구조
- PWA 설정
- `package.json`

원칙:

```text
어디로 가는지, 무엇을 저장하는지, 어떤 데이터를 부르는지는 바꾸지 않는다.
어떻게 보이는지만 바꾼다.
```

## 8. BottomTabBar 작업 규칙

BottomTabBar는 공통 컴포넌트이므로 특히 조심한다.

유지해야 할 것:

- `useNavigate`
- `useLocation`
- `handleAdd()` 라우팅 분기
- 각 버튼의 `navigate()` 경로
- `isActive()` 활성 상태 판단 방식

구현 기준:

- `fixed bottom-0 left-1/2 -translate-x-1/2 w-full max-w-[448px]`
- Figma의 고정 좌표가 아니라 flex 기반으로 구성
- 아이콘은 Figma SVG path를 기준으로 인라인 React 컴포넌트로 관리
- active/inactive 색상은 `currentColor` 또는 토큰 클래스로 처리
- 중앙 추가 버튼도 Figma 시안대로 하되, 여러 폭에서 레이아웃이 깨지지 않게 구현
- 외부 이미지 URL 대신 inline SVG를 우선 사용

유지보수 구조:

- 탭 정보가 반복되면 파일 내부에 배열로 정리해도 된다.
- 반복 JSX는 파일 내부의 작은 `TabButton` 컴포넌트로 분리해도 된다.
- BottomTabBar 하나를 위해 과한 폴더 구조나 새 abstraction을 만들지 않는다.
- 새 파일은 명확히 재사용 가치가 있을 때만 만든다.

## 9. 아이콘 / SVG 규칙

- 새 아이콘 라이브러리를 추가하지 않는다.
- Figma SVG path를 임의로 다른 아이콘으로 대체하지 않는다.
- active/inactive 색상 변경이 필요한 SVG는 `currentColor`를 우선 사용한다.
- SVG 크기는 Figma 값을 참고하되, 버튼 영역은 모바일 폭에서 깨지지 않게 조정한다.
- Figma asset URL로 이미지를 직접 불러오는 방식은 피한다.

## 10. 레이어드 아키텍처 기준

이 프로젝트에서는 복잡한 아키텍처보다 역할 분리를 우선한다.

권장 구조:

- `pages/`: 라우팅되는 화면과 화면 조립
- `components/`: 재사용 UI 컴포넌트
- `api/`: 서버 통신
- `context/`: 전역 상태
- `index.css`: 디자인 토큰과 전역 레이아웃 기준

컴포넌트 내부 구조:

- 화면 동작 로직과 UI 반복 구조를 구분한다.
- 반복되는 UI는 작은 내부 컴포넌트로 분리한다.
- 반복되는 데이터는 배열 상수로 정리한다.
- 단일 파일에서 충분히 관리 가능한 것은 새 파일로 빼지 않는다.
- 새 폴더나 파일은 실제 재사용성이 있을 때만 만든다.

## 11. 검증 규칙

수정 후 가능한 범위에서 확인한다.

- `npm run dev`로 실행
- 브라우저에서 해당 화면 확인
- 360px, 393px, 430px, 448px 폭 확인
- BottomTabBar가 화면 하단에 고정되는지 확인
- 텍스트, 아이콘, 버튼이 겹치거나 잘리지 않는지 확인
- 라우팅이 기존처럼 동작하는지 확인

테스트나 실행이 불가능하면 최종 응답에 이유를 짧게 남긴다.

## 12. 응답 방식

작업 전:

- 필요한 파일을 읽고 바로 수정한다.
- 사용자가 표를 요청하지 않으면 표를 만들지 않는다.

작업 중:

- 무엇을 수정하는지 짧게 알려준다.
- 불확실하거나 위험한 동작 변경이 필요할 때만 질문한다.

작업 후:

- 바뀐 파일과 핵심 변경만 짧게 요약한다.
- 실행/화면 확인 여부를 말한다.
- 사용자에게 복잡한 프론트 판단을 요구하지 않는다.

## 13. 기존 버그 발견 시

작업 중 기존 버그나 이상한 코드를 발견해도 자동으로 고치지 않는다.

- 현재 요청과 직접 관련 있으면 짧게 보고한다.
- 같이 고쳐도 되는지 물어본다.
- 사용자가 명시적으로 승인한 것만 수정한다.

## 14. 코드 스타일

- 기존 코드 스타일을 따른다.
- 주석은 꼭 필요할 때만 짧게 작성한다.
- 불필요한 주석, 설명용 주석은 추가하지 않는다.
- 이모지는 코드에 넣지 않는다.
- 새 의존성은 추가하지 않는다.
